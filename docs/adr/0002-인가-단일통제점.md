# ADR 0002 — 인가 단일 통제점 (Authorization Single Choke Point)

- **상태**: 채택 (Accepted)
- **날짜**: 2026-06-11
- **불변식**: INV-1(권한 상승 0), INV-7(역할 위계)
- **상세 설계**: `docs/grok-discord-bot-권한설계.md`

## 맥락

디스코드 관리 봇은 서버 관리 권한을 통째로 보유한다. 이 권한이 호출자에게 그대로 위임되면, 권한 없는 일반 유저가 봇을 시켜 채널을 닫거나 다른 유저를 차단할 수 있다(혼동된 대리자, confused deputy).

두 개의 공격 표면이 있다:
1. **슬래시 명령** — 핸들러에 권한 체크가 없으면 뚫림.
2. **자연어(Function Calling)** — "@봇 저 사람 밴해줘". 슬래시에만 체크를 걸면 **이 경로로 우회**된다(가장 위험).

## 결정

1. **단일 인가 게이트**: 슬래시·자연어를 불문하고 **모든 관리/모더레이션/채널/유저 액션은 `auth.authorize()` 하나를 반드시 통과**한다.
2. **요청자 권한으로 행동**: 액션 허용은 *봇이 무엇을 할 수 있는가*가 아니라 *요청한 사람(`ctx.requester`)이 그 권한을 가졌는가*로 판정한다. 봇의 권한으로 행동하지 않는다.
3. **함수 호출 ≠ 권한**: 모델이 `ban_user`를 호출했다는 사실이 권한을 부여하지 않는다. 권한은 항상 요청자로부터 재도출한다.
4. **기본 거부**: `ACTIONS` 권한 매트릭스에 등록되지 않은 액션은 비관리자에게 거부가 기본값. 새 관리 함수는 스펙 등록 없이 머지 금지.
5. **티어 모델**: MEMBER < STAFF < ADMIN < OWNER. Discord 네이티브 권한(`Administrator` 등)으로 무설정 동작 + 역할(`staff/admin_role_ids`) 설정으로 세밀화.
6. **다층 방어**: L0 Discord `default_member_permissions`(슬래시 1차) → L1 봇/웹훅 차단 → L2 `authorize()` → L3 역할 위계 → L4 감사 로그.
7. **경계 강제**: `auth`는 순수 판정(nsfw/moderation/tools를 import하지 않음). 관리 액션의 인가 우회 직접 실행은 lint(eslint-plugin-boundaries)와 코드리뷰로 차단.

## 대안

- **슬래시 명령에만 권한 체크**: 자연어 경로가 그대로 뚫림 → 기각.
- **봇 권한으로 실행 + 사후 검사**: 혼동된 대리자 그대로 → 기각.
- **역할 화이트리스트만(네이티브 권한 무시)**: 무설정 서버에서 동작 불가 → 하이브리드로 절충.

## 결과

- M4에서 `authorize()`, `ACTIONS`, `resolveTier`, 거부 메시지, 감사 적재를 구현.
- M4 게이트: 일반 유저 관리 명령 거부율 100%, **슬래시↔자연어 판정 일치율(parity) 100%**, 봇/웹훅 거부, 미등록 액션 거부.
- M5(서버 관리 기능)는 이 게이트 위에만 올린다.
- 새 관리 함수 추가 시 `ACTIONS` 등록을 PR 체크리스트 필수 항목으로.
