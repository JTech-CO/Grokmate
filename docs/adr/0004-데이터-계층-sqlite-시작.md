# ADR 0004 — 데이터 계층: SQLite 시작 + 스토어 추상화

- **상태**: 채택 (Accepted)
- **날짜**: 2026-06-13
- **관련 미결 질문**: Q5(초기 SQLite vs PostgreSQL 바로)
- **관련**: ADR 0001(스택), 기술백서 §9(스키마)·§10(메모리), 권한설계 §7

## 맥락

M2(영속화)를 시작한다. 운영 타깃은 PostgreSQL + Redis다(ADR 0001, Q7의 24/7 가동). 그러나 1인 개발 초기 단계이고, 개발 머신이 Windows라 **네이티브 Redis가 없다**(Memurai/WSL/Docker 필요). 처음부터 Postgres+Redis를 로컬에 띄우면 인프라 설치·구동 비용이 M2~M6 개발 속도를 떨어뜨린다. 데이터 계층을 "지금 무엇으로 시작하고, 나중에 어떻게 갈아끼울지" 정한다.

## 결정

1. **영속은 SQLite로 시작**(`better-sqlite3`). ORM은 **Drizzle** — SQLite와 PostgreSQL을 같은 코드 스타일로 지원해 추후 전환 비용을 낮춘다.
2. **데이터 접근은 리포지토리 뒤에 숨긴다.** 상위 모듈(core/auth/nsfw 등)은 테이블이 아니라 `db`가 노출하는 리포지토리 함수만 호출한다. 스키마·마이그레이션은 Drizzle(`drizzle-kit`)로 관리.
3. **컨텍스트 스토어는 인터페이스로 추상화.** 초기 구현은 프로세스 인메모리(맵 기반), 후속에 Redis 백엔드로 스왑. M1의 인메모리 대화 윈도우를 이 인터페이스로 흡수한다.
4. **경계 유지**: `db`는 `config`만 의존한다(CLAUDE.md §2.1). DB 드라이버·Drizzle 인스턴스 생성은 `db` 안에서만. 시크릿/경로는 `config` 경유.

## 근거

- 인프라 0 설치로 M2 이후 기능 개발을 막힘 없이 진행한다.
- Drizzle 추상화 + 리포지토리 경계로 **Postgres 전환을 "방언·마이그레이션 교체 + 회귀 테스트" 수준으로 국소화**한다.
- 스토어 인터페이스로 **Redis 도입(M8/배포)을 한 파일 교체**로 가둔다.
- 개인정보 최소 수집(기술백서 §9): 대화 원문 장기 보관 지양, 요약·메타 위주. 감사 테이블(`tool_call_audit`·`moderation_log`)은 전수 기록(INV-1 추적성).

## 대안

- **PostgreSQL + Redis 바로**: 운영과 동일하나 초기 인프라 비용 + Windows Redis 부재 → 보류. 전환 트리거: 다중 인스턴스/샤딩, 동시성·동시 쓰기 경합, 배포(M8).
- **Prisma**: 무난하나 런타임·생성 단계가 무겁다. Drizzle이 경량·SQL 친화·다중 방언이라 채택.
- **파일/JSON 영속**: 쿼리·마이그레이션·동시성에서 열위 → 기각.

## 결과

- 의존성: `drizzle-orm`, `better-sqlite3` (+ dev `drizzle-kit`). SQLite 파일 경로는 `config`(`DB_PATH`, 기본 `./data/grokmate.db`), `.gitignore`에 데이터 파일 제외.
- M2 게이트(하네스 §1 M2): 마이그레이션 적용·롤백 / 핵심 CRUD 왕복(특히 `tool_call_audit`·`nsfw_consent`) / 타입체크 그린 / 컨텍스트 스토어 저장→재로드 동일.
- **Postgres/Redis 전환은 별도 ADR** + INV 회귀 테스트로 가드.
- 보존 기간·삭제 요청(개인정보) 정책은 후속(M7 연령인증·운영)에서 구체화, 스키마엔 타임스탬프·메타로 대비.
